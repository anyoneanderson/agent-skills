# 抽象的な処理説明の検査

仕様書を読み込んだ後、Check 18 として実行する。
抽象的な処理の Pattern は `spec-writing/references/abstract-verbs.ja.md` だけから読み、`spec-inspect` では二重管理しない。

## 入力

- `requirement.md`、`design.md`、`tasks.md`。
- 存在する場合は `test.md`。
- `spec-writing` から選んだ日本語の抽象動詞表。

## 検査候補の選択

1. 各仕様書の本文、表、箇条書き、Mermaid のメッセージを走査する。
2. 語彙表の `Pattern` と一致した行を候補にする。
3. 通常のコードブロック、ログ、設定例、Pattern を文字列の一部に含むだけの識別子は除外する。
4. Mermaid の `sequenceDiagram` は検査対象にする。
   隣接する `Note` はメッセージの文脈として扱う。
5. Pattern との一致だけでは finding を作らない。

## 証拠を探す範囲

候補ごとに、同じ行、候補を含む箇条書きまたは段落、直前直後の文を読む。
シーケンス図では、候補のメッセージと近接するメッセージまたは注記を読む。

`spec-writing` が定義する次の4点を集める。

1. 処理を実行する主体。
2. 処理を始める開始条件、状態、イベント、または入力。
3. 保存、比較、計算、送信、拒否、停止など、実装で観測できる動作。
4. 結果を受け取る保存先または送信先。

候補行の `Required evidence` も適用する。
複数の文から4点を一意に追える場合は合格とする。
数学上の操作は、入力、変換規則、出力が定義されていれば合格とする。

## WARNING の判定

4点のうち一つ以上が不足し、実装者が具体的な処理を一意に決められない場合だけ WARNING を作る。
近接する記述に証拠があれば報告しない。
同じ文が引用と本文に重複していても、同じファイルと行に対する finding は一件にまとめる。

## finding の契約

既存のレポート形式で次の項目を保存する。

```text
ID: WARNING-{seq}
Title: 抽象的な処理説明: {pattern}
File: {filename}
Line: {line_number}
Description: {引用文} だけでは {missing_elements} を判断できない
Suggestion: {主体、開始条件または入力、観測できる動作、結果の渡し先を補った修正文}
Source rule: {AV-XXX}
```

`missing_elements` には不足する項目だけを列挙する。
仕様書から確定できる動作は保持する。
値を確定できない場合は、主体、開始条件、動作、渡し先を作らず、角括弧で名前つきのプレースホルダーを示す。

## 語彙の読込エラー

`spec-writing`、対象言語の語彙ファイル、必須列、安定 ID のいずれかを読めない場合は、Pattern 一覧を内蔵または推測して代替しない。
「抽象処理説明の検査を実行できなかった」と、不足または破損した参照を一件の WARNING に記録し、Check 1 から Check 17 を続行する。
