# Test Plan Phase - 受け入れテスト計画の生成

## 概要

受け入れテスト計画（test.md）を生成するフェーズ。
requirement.md・design.md・tasks.md を入力として、完成した実装に対して評価側が
実行するブラックボックスのチェックリストを作る。

**test.md を tasks.md と分ける理由**: 2つのファイルは読み手が違う。tasks.md は
作る側（spec-code）が機能を組み立てるために読み、test.md は壊す側（spec-evaluate
のような受け入れ評価者）が検証するために読む。分けておくことで、評価者は作る側の
思い込みを引き継がず、要件だけを起点に検証できる。

**重要**: このフェーズは計画の「生成」のみ。テストの実行は行わない。

## 実行手順

### 1. 仕様3点セットの特定

```bash
find .specs -name "requirement.md" -type f
find .specs -name "design.md" -type f
```

仕様セットから以下を抽出:
- すべての機能要件ID（`REQ-XXX`）
- すべての非機能要件ID（`NFR-XXX`）
- design.md に書かれた受け入れに関わる振る舞い

### 2. カバレッジ原則

1. すべての `REQ`・`NFR` は最低1つのテストケースから参照されること。
   テストケースのない要件はカバレッジの漏れ — ケースを追加する。
2. 1テストケースは1つの観測可能な振る舞いを検証する。複数の振る舞いを持つ要件は
   複数ケースに分割する（`T-A01`, `T-A02`, ...）。
3. 仕様セットにない振る舞いをでっち上げない。要件が現状のままでは検証不能なら、
   手順を捏造せず注記として記録する。

### 3. テストケースID体系

- 接頭辞は `T-A`（Acceptance）: `T-A01`, `T-A02`, ...
- `T-A` 接頭辞は tasks.md の `T-`（実装タスク）IDと衝突しない。両ファイルを
  相互参照しても曖昧にならない。

### 4. 検証方法

各ケースは検証方法を必ず1つだけ宣言する:

| 方法 | 意味 | 必要なもの |
|------|------|-----------|
| `playwright` | 実ブラウザで UI を操作し結果を検証する | アプリ起動レシピ（起動コマンド・URL） |
| `command` | シェルコマンドを実行し終了コード／出力を検証する | 実行コマンドと期待終了コード |
| `file-check` | 成果物の存在・内容を検証する | パスと期待する内容の目印 |

- `file-check` は生成した成果物に使う。ソースや設定の生ファイルから確認できるのは
  文字列そのものを要求する要件だけであり、有効な設定値や実行時の振る舞いは確認できない。
  詳細は第5節を参照する。
- `playwright` ケースはアプリ起動レシピに依存する。起動レシピが無い場合でも
  ケースは書くが、実行にレシピが必要であることを明記する。
- 検証方法フィールドは全ケースで**必須**。検証方法が空のケースは実行不能であり
  出力してはならない。
- 検証コマンドは対象環境でそのまま実行できるものに限る。シェル方言・PCRE 拡張
  （例: `grep -E` の否定先読み。POSIX ERE では構文エラー）・言語バージョン依存の
  API（例: Python 3.10 で追加された `sys.stdlib_module_names`）を避ける。避けられ
  ない場合は、必要なバージョンやツールをケースの `コマンド` 欄に前提として明記する。

- `command` ケースのコマンドは生成時に実行可能性を実確認する: 実際に実行（または
  同等の dry run）し、ケースが主張する内容を本当に検査していることを確かめる。
  場当たりな起動形より既存の `package.json` script を優先する。特に引数の受け渡しに
  注意する — 例えば `pnpm test -- --coverage` は `vitest run -- --coverage` に展開され、
  `--coverage` が位置引数に格下げされる。exit 0 のまま coverage レポートを一切生成せず、
  「何も検証しないのに合格する」ケースになる。

### 5. 証拠基盤と脅威モデル

各ケースの手順とコマンドを書く前に、次を決める。

1. 検証する主張と脅威モデルを定義する。
   受け入れ検査は、記載した検証コマンドとテスト機構がそのまま実行される前提で、実装の正しさを判定する。
   この境界が合否の解釈に影響する場合は、ケースの`手順`または`期待結果`に明記する。
2. 意味上の主張には、入力の評価を担うシステムまたはツールが生成した証拠を使う。
   - **計測の実出力**: 検証対象の動作を実行して生成した coverage レポート、レスポンス、クエリ結果など。
   - **評価済みオブジェクト**: 対象のランタイムまたはツールが、サポートするインターフェースを介して設定やモジュールを読み込んだ後の値。
3. `grep`、正規表現、部分文字列の一致を使って、ソースや設定の生テキストから有効な設定値や実行時の振る舞いを推定しない。
   要件自体が生成ファイルの必須ヘッダなどの文字列を指定する場合に限り、字句検査を使ってよい。
4. テストの無効化、`skip`または`todo`の追加、計測対象からの除外、検証コマンドの書き換えなど、意図的な検査回避はコードレビューゲートで検出する。
   受け入れコマンドへ字句検査を重ねて、これらの回避を検出しようとしない。
   仕様が改ざん耐性や検査回避の検出を明示的に要求する場合は、そのセキュリティ上の振る舞いを独立した受け入れ条件として検証する。

### 6. 出力形式

```markdown
# 受け入れテスト計画 - {feature}
type: test-plan

## T-A01: [REQ-001] ログインできる
- 手順: 1. {url}/login を開く  2. 有効な認証情報を入力  3. 送信
- 期待結果: ダッシュボードへ遷移し、ユーザー名が表示される
- 検証方法: playwright       # playwright | command | file-check
- コマンド: -               # command のとき: 実行コマンドと期待終了コード

## T-A02: [NFR-001] レスポンスが 500ms 以内に返る
- 手順: 1. GET /api/items を10回送信
- 期待結果: p95 レイテンシ < 500ms
- 検証方法: command
- コマンド: `hey -n 10 http://localhost:3000/api/items` → exit 0, p95 < 500ms

## T-A03: [REQ-005] エクスポートファイルが書き出される
- 手順: 1. 設定画面からエクスポートを実行
- 期待結果: exports/ 配下に CSV ファイルが作成される
- 検証方法: file-check
- コマンド: exports/report.csv が存在し、ヘッダ行を含む
```

- フィールド構成は固定: `手順` / `期待結果` / `検証方法` / `コマンド`。
- `playwright`・`file-check` ケースでは `コマンド` フィールドに期待する成果物や
  検証内容を書く。情報が無いときのみ `-` を使う。

### 7. 保存先

```
.specs/[project-name]/test.md
```

## Full ワークフローとの統合

test.md は full ワークフローの**最終ステップ**で、requirement.md・design.md・
tasks.md がそろった後に生成する。3点セットを読むが変更はしない — 既存の3点セット
出力は変わらない。

## 品質チェックリスト

生成後の確認項目:

1. [ ] `type: test-plan` ヘッダがある
2. [ ] requirement.md の全 `REQ`・`NFR` が最低1ケースから参照されている
3. [ ] 全ケースIDが `T-A` 接頭辞を使っている
4. [ ] 全ケースに空でない `検証方法`（playwright / command / file-check）がある
5. [ ] `playwright` ケースがアプリ起動依存を明記している
6. [ ] 仕様セットにない振る舞いを作り出したケースがない
7. [ ] 全検証コマンドが対象環境でそのまま実行できる — シェル方言・PCRE・バージョン
   依存 API の前提がない、または前提として明記されている
8. [ ] 全 `command` ケースを生成時に実行（または dry run）し、主張どおりの検査が
   行われることを確認した — 引数の受け渡しを含む
9. [ ] 意味上の主張は計測の実出力または評価済みオブジェクトで検証し、ソースや設定の
   生テキストに対する字句検査は、文字列そのものを要求する場合だけに限定している
10. [ ] 脅威モデルの境界が合否の解釈に影響するケースでは、その境界を`手順`または
    `期待結果`に明記し、意図的な検査回避は、仕様が検出を要求する場合を除いて
    コードレビューゲートの責務としている
