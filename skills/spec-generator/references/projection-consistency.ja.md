# 投影整合パス

初回生成とすべての仕様修正の後、仕様書の完了を報告する前にこの手順を実行する。
投影更新の正本はこのファイルとし、workflow plannerなどの呼び出し側はチェックリストを
複製せず、この手順を呼び出す。

## 変更した事実の集合

編集前に、追加、削除、変更する事実を列挙する。

- Requirement ID、設計判断、数値基準、方式名
- requirementまたはdesign本文が意味を定める用語
- 要求、その実装タスク、受け入れテストの関係

各事実の変更前と変更後の表記を仕様書一式から検索する。
検索結果は投影候補であり、同じ意味を表す証拠ではない。

## 投影先の一覧

変更した事実ごとに、存在する投影先を確認する。

| ID | 投影先 |
|----|--------|
| PG-001 | 現在の本文を要約する§0、§12、見出し。§12は存在する場合の末尾の要約または結論セクションを指す |
| PG-002 | §1トレーサビリティ表と、Requirement IDを列挙する箇所 |
| PG-003 | 設計判断を定義するrequirementとdesignの本文 |
| PG-004 | 影響を受ける要求または設計を実装する`tasks.md`のタスク |
| PG-005 | 影響を受ける要求を検証する`test.md`のテストケース |

変更元と影響を受けるすべての投影先を同じ編集内で更新する。
古い投影先を後続の修正へ残さない。

## 検証

1. `requirement.md`、`design.md`、`tasks.md`、`test.md`から、定義したRequirement IDと
   参照したRequirement IDを再抽出する。未定義の参照と、必要な下流投影が消えた要求を
   拒否する。
2. 同じ事実を表すID、数値基準、方式名などの構造化された値を完全一致で比較する。
3. 要約文と本文は意味で比較する。文字列の完全一致を意味の一致の代用にしない。
4. 廃止したID、値、名称を検索し、残った箇所を意図した履歴や例示、または古い投影に
   分類する。
5. 変更した要求または設計判断に対応する実装タスクと受け入れテストがあることを確認する。
   投影先が不要なら、その理由を明記する。

影響を受ける要約、トレーサビリティ行、タスク、テストのいずれかが古い事実を示す間は、
完了を報告しない。
